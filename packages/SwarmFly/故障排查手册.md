## 一、故障分类与响应

### 1.1 故障级别定义

| 级别 | 名称 | 描述 | 响应时间 | 示例 |
|------|------|------|----------|------|
| P0 | 严重故障 | 系统不可用、数据丢失风险 | 5分钟 | 全部服务宕机 |
| P1 | 重大故障 | 核心功能受损 | 15分钟 | 规则引擎不可用 |
| P2 | 一般故障 | 非核心功能异常 | 1小时 | 监控面板加载慢 |
| P3 | 轻微问题 | 可接受的性能/体验问题 | 工作时间 | 日志格式问题 |

### 1.2 故障响应流程

```
P0/P1故障:
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ 发现故障 │ → │ 通知值班 │ → │ 快速止血 │ → │ 根因分析 │
└─────────┘    └─────────┘    └─────────┘    └─────────┘
                                                       ↓
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ 复盘总结 │ ← │ 上线验证 │ ← │ 问题修复 │ ← │ 影响评估 │
└─────────┘    └─────────┘    └─────────┘    └─────────┘
```

## 二、常见故障排查

### 2.1 规则引擎故障

#### 症状: 规则执行失败率飙升

**排查步骤:**

```bash
# 1. 检查规则引擎健康状态
curl -s http://localhost:8080/health | jq

# 2. 查看最近错误日志
kubectl logs -f deployment/swarmfly-fly2 --tail=100 | grep ERROR

# 3. 检查Redis连接
redis-cli -h redis.swarmfly.svc.cluster.local ping

# 4. 查看规则执行队列深度
curl -s http://localhost:8080/metrics | grep fly2_rule_execution
```

**常见原因:**

| 原因 | 症状 | 解决方案 |
|------|------|----------|
| Redis连接超时 | 缓存操作失败 | 检查Redis网络、增大连接池 |
| 规则语法错误 | 特定规则失败 | 回滚到上一个版本 |
| 并发过高 | 队列堆积 | 扩容或限流 |
| 死锁 | 服务hang住 | 重启并检查资源竞争配置 |

**快速止血:**

```bash
# 1. 禁用问题规则
curl -X POST http://localhost:8080/api/v1/rules/{rule_id}/disable

# 2. 临时扩容
kubectl scale deployment swarmfly-fly2 --replicas=5

# 3. 切换到降级模式
curl -X PUT http://localhost:8080/api/v1/config/degradation-mode \
  -d '{"mode": "degraded"}'
```

#### 症状: 规则执行延迟过高

```bash
# 1. 查看延迟分布
curl -s http://localhost:8080/metrics | grep fly2_rule_execution_duration
```

```python
# 如果缓存命中率低

RULE_OPTIMIZATION = {
    "cache_intermediate_results": True,
    "parallel_condition_eval": True,
    "early_termination": True
}
```

### 2.2 趋势分析故障

#### 症状: 预测准确度下降

**排查步骤:**

```bash
# 1. 检查预测模型状态
curl -s http://localhost:8081/api/v1/models/status

# 2. 查看数据源健康
curl -s http://localhost:8081/api/v1/datasources/health

# 3. 分析预测误差分布
curl -s http://localhost:8081/api/v1/predictions/error-analysis?period=24h
```

**常见原因与解决:**

| 原因 | 诊断方法 | 解决方案 |
|------|----------|----------|
| 数据质量下降 | 检查数据源延迟 | 修复数据管道 |
| 模型过期 | 检查模型最后更新时间 | 触发模型重训练 |
| 异常数据干扰 | 查看离群点比例 | 调整异常值处理 |
| 季节性变化 | 分析误差周期 | 更新训练数据 |

**模型重训练:**

```bash
# 触发手动重训练
curl -X POST http://localhost:8081/api/v1/models/{model_name}/retrain \
  -H "Content-Type: application/json" \
  -d '{
    "training_data_range": "90d",
    "validation_split": 0.2,
    "notify_on_complete": true
  }'
```

### 2.3 工具中心故障

#### 症状: 消息发布失败

**排查步骤:**

```bash
# 1. 检查RabbitMQ状态
kubectl exec -it rabbitmq-0 -- rabbitmqctl status

# 2. 查看队列深度
kubectl exec -it rabbitmq-0 -- rabbitmqctl list_queues name messages

# 3. 检查发布者确认
curl -s http://localhost:8082/metrics | grep fly5_mq_publish
```

**快速止血:**

```bash
# 1. 切换到备节点
kubectl patch service rabbitmq -p '{"spec":{"selector":{"app":"rabbitmq-backup"}}}}'

# 2. 启用消息缓冲
curl -X PUT http://localhost:8082/api/v1/config/mq-buffer \
  -d '{"enabled": true, "max_size": 10000}'

# 3. 降级到同步模式（不推荐长期使用）
curl -X PUT http://localhost:8082/api/v1/config/mq-mode \
  -d '{"mode": "sync", "timeout_ms": 5000}'
```

---

#### 症状: 资源池耗尽

**排查步骤:**

```bash
# 1. 查看资源池状态
curl -s http://localhost:8082/api/v1/resources/pool/status

# 2. 检查资源分配情况
curl -s http://localhost:8082/api/v1/resources/allocations?status=active

# 3. 查看资源泄漏
curl -s http://localhost:8082/api/v1/resources/leaks?threshold=1h
```

**解决方案:**

```bash
# 1. 紧急扩容
kubectl scale deployment swarmfly-fly5 --replicas=10

# 2. 清理过期资源
curl -X POST http://localhost:8082/api/v1/resources/cleanup \
  -d '{"older_than": "1h", "force": true}'

# 3. 设置资源上限告警
curl -X PUT http://localhost:8082/api/v1/config/resource-alerts \
  -d '{"threshold": 0.8, "action": "auto-scale"}'
```

### 3.2 日志分析

```python
# 日志分析脚本
import re
from collections import Counter

def analyze_error_logs(log_file: str) -> Dict:
    """分析错误日志"""
    error_pattern = r'ERROR.*?(\w+Exception): (.+?)(?=\n|$)'
    
    errors = []
    with open(log_file) as f:
        for line in f:
            match = re.search(error_pattern, line)
            if match:
                errors.append({
                    "type": match.group(1),
                    "message": match.group(2),
                    "timestamp": extract_timestamp(line)
                })
    
    return {
        "total_errors": len(errors),
        "error_types": Counter(e["type"] for e in errors),
        "top_errors": Counter(f"{e['type']}: {e['message']}" for e in errors).most_common(10),
        "time_distribution": group_by_time(errors)
    }
```

### 4.2 故障转移流程

```python
# 自动故障转移
class AutomaticFailover:
    """自动故障转移"""
    
    async def execute_failover(self, service: str, reason: str):
        """执行故障转移"""
        # 1. 记录故障事件
        await self.record_failure(service, reason)
        
        # 2. 触发备用服务
        if service == "fly2-primary":
            await self.activate_standby("fly2-standby")
        elif service == "rabbitmq-primary":
            await self.promote_standby("rabbitmq-standby")
        
        # 3. 更新路由
        await self.update_routing(service)
        
        # 4. 通知相关方
        await self.notify_failure_handlers(service, reason)
        
        # 5. 启动根因分析
        await self.trigger_rca(service)
```
