## 一、错误码概述

SwarmFly 系统采用统一的错误码体系，便于问题定位和排查。

### 1.1 错误码格式

```
FLY-[层级]-[模块]-[编号]
```

| 组成部分 | 说明 | 示例 |
|---------|------|------|
| FLY | 系统标识 | FLY |
| 层级 | 2-法则层 / 3-趋势层 / 5-工具层 | 2 / 3 / 5 |
| 模块 | 具体模块缩写 | RE(规则) / TR(趋势) / MQ(队列) |
| 编号 | 四位数字 | 0001 |

### 1.2 错误等级

| 等级 | 标识 | 说明 | 影响范围 |
|------|------|------|----------|
| P0 | CRITICAL | 致命错误，系统不可用 | 整个系统 |
| P1 | ERROR | 严重错误，功能受损 | 部分功能 |
| P2 | WARNING | 警告，潜在问题 | 功能正常但需关注 |
| P3 | INFO | 提示信息 | 无实际影响 |

## 六、错误日志规范

### 6.1 日志格式

```json
{
  "timestamp": "2026-06-08T10:30:00.000Z",
  "level": "ERROR",
  "error_code": "FLY-3-PR-0001",
  "message": "Prediction model error",
  "context": {
    "metric_name": "cpu_usage",
    "model": "linear",
    "data_points": 5
  },
  "trace_id": "abc123",
  "request_id": "req-456"
}
```

### 6.2 错误处理最佳实践

1. **分级处理**: 根据错误等级采取不同策略
   - P0: 立即告警，触发应急响应
   - P1: 记录并告警，尝试自动恢复
   - P2: 记录日志，监控趋势
   - P3: 记录debug日志

2. **错误上报**: 关键错误需上报到监控系统

3. **错误恢复**: 每次错误处理需考虑恢复路径

## 七、错误码维护

### 7.1 新增错误码流程

1. 在对应模块的文档中新增条目
2. 在代码中定义错误码常量
3. 更新单元测试覆盖新错误码
4. 通知相关团队

### 7.2 错误码废弃流程

1. 在错误码文档中标记为废弃
2. 在代码中保留但标记为deprecated
3. 在下一主版本中移除
